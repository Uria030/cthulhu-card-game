# 第八部分：招式池管理

## 8.1 招式池在變體編輯區中的位置

招式池是變體編輯區的**核心區塊之一**，位於區塊四（AI 行為配置）之後、
區塊六（敘事狀態描述）之前。
標題：「⚔ 招式池」

## 8.2 招式池的兩層結構

每隻怪物的可用招式來自兩個來源：

```
種族級共享招式（species_id 不為 NULL, variant_id 為 NULL）
  ↓ 所有變體都可使用
  ＋
變體專屬招式（variant_id 不為 NULL）
  ↓ 只有該變體可使用
  ＝
該變體的完整招式池
```

前端顯示時：
- 種族級共享招式用**淡金色底色** + 「共享」標籤
- 變體專屬招式用**標準底色**
- 種族級共享招式的編輯按鈕跳轉到種族編輯的共享招式區

## 8.4 招式卡編輯表單

點擊「編輯」或「新增專屬招式」展開行內編輯表單：

### 基本資訊行

| 欄位 | 類型 | 說明 |
|------|------|------|
| 代碼 (code) | 文字輸入 | `snake_case` |
| 中文名 (name_zh) | 文字輸入 | e.g. '利爪撕裂' |
| 英文名 (name_en) | 文字輸入 | e.g. 'Claw Strike' |

### 戰鬥數值行

| 欄位 | 類型 | 說明 |
|------|------|------|
| 防禦屬性 | 下拉選單 | 七大屬性之一 — 調查員用這個屬性防禦 |
| DC 覆寫 | 數字輸入 | 留空 = 使用變體的 DC |
| 物理傷害 | 數字輸入 | |
| 恐懼傷害 | 數字輸入 | |
| 傷害元素 | 下拉選單 | physical / fire / ice / electric / arcane |
| 權重 | 數字輸入 | 預設 10，影響被抽到的機率 |

### 附加效果行

| 欄位 | 類型 | 說明 |
|------|------|------|
| 施加狀態 | 可編輯列表 | [狀態▼] [數值] [持續回合] [觸發時機: 命中/防禦失敗] |
| 特殊效果 | 文字區域 | JSON 格式，自由填寫（進階用） |

### 觸發條件行（可選）

| 欄位 | 類型 | 說明 |
|------|------|------|
| 觸發條件類型 | 下拉選單 | 無 / 目標狀態 / HP 低於百分比 / 戰鬥階段 |
| 條件參數 | 動態欄位 | 依類型顯示不同輸入 |

### 三段敘事行

| 欄位 | 類型 | 說明 |
|------|------|------|
| 攻擊敘事 | 文字區域 | 怪物發動攻擊的描述 |
| 命中敘事 | 文字區域 | 調查員防禦失敗的描述 |
| 閃避敘事 | 文字區域 | 調查員防禦成功的描述 |

### 防禦屬性選擇的設計引導

選擇防禦屬性時，下方顯示該怪物家族的「防禦屬性傾向」作為參考：

```
家族傾向：力量 ■■■ 高  |  體質 ■■■ 高  |  敏捷 ■■ 中  |  意志 ■ 低
```

這是設計引導，不是強制限制。

## 8.5 招式池設計建議

變體編輯區底部顯示招式池設計建議（根據位階）：

| 位階 | 建議招式卡數量 | 說明 |
|------|-------------|------|
| 雜兵 | 2–3 張 | 簡單直接，1–2 種防禦屬性 |
| 威脅 | 3–4 張 | 開始有變化，2–3 種防禦屬性 |
| 精英 | 4–6 張 | 需要策略應對，3–4 種防禦屬性 |
| 頭目 | 6–8 張 | 豐富多變，含階段限定招式 |
| 巨頭 | 8–10 張 | 極度豐富，含傳奇行動關聯招式 |

當前招式卡數量未達建議下限時，顯示提示：
`⚠ 此變體目前有 2 張招式卡，建議至少 4 張（精英級）`

## 8.6 招式池防禦屬性分佈視覺化

在招式池區塊頂部，顯示一個小型橫向條狀圖，
展示當前招式池中各防禦屬性的比例：

```
防禦屬性分佈：
力量 ████████ 40%  (2 張)
體質 ████     20%  (1 張)
敏捷 ████     20%  (1 張)
意志 ████     20%  (1 張)
```

這讓設計師一眼看出是否過度集中於某個屬性。

# 第九部分：Gemini AI 生成

## 9.1 三種 AI 生成模式

| 模式 | 觸發位置 | 功能 |
|------|---------|------|
| 單張招式卡生成 | 招式卡編輯表單內 | 根據怪物資訊生成一張招式卡的完整資料 |
| 批次招式池生成 | 招式池區塊底部 | 根據位階建議數量一次生成整個招式池 |
| 敘事描述生成 | 敘事狀態描述區塊 | 根據怪物資訊生成 5 階段敘事描述 |

## 9.2 Gemini Prompt — 單張招式卡生成

```
你是一個克蘇魯神話 TRPG 的怪物招式設計師。
請為以下怪物設計一張招式卡。

## 怪物資訊
- 名稱：{variant.name_zh}（{variant.name_en}）
- 種族：{species.name_zh}
- 家族：{family.name_zh}
- 位階：{tier_name}（DC {variant.dc}）
- 攻擊元素：{resolved.attack_element}
- 家族主題：{family.theme_zh}

## 家族防禦屬性傾向
{family.defense_attribute_tendency 展開為文字}

## 已有的招式（避免重複）
{現有招式列表}

## 輸出格式（嚴格 JSON）
{
  "name_zh": "招式中文名",
  "name_en": "招式英文名",
  "defense_attribute": "七大屬性之一",
  "damage_physical": 數字,
  "damage_horror": 數字,
  "damage_element": "元素代碼",
  "inflicts_status": [{"code": "狀態代碼", "value": 數字, "duration": 回合數, "on": "hit"}],
  "weight": 10,
  "narrative_attack_zh": "攻擊敘事",
  "narrative_hit_zh": "命中敘事",
  "narrative_miss_zh": "閃避敘事"
}

## 設計原則
1. 防禦屬性遵循家族傾向但不必完全一致
2. 傷害值參考位階範圍：{tier_damage_range}
3. 敘事要有克蘇魯神話的恐怖氛圍
4. 每張招式卡要有獨特的戰鬥場景感
5. 避免與已有招式在防禦屬性和場景上重複
```

## 9.3 Gemini Prompt — 批次招式池生成

```
你是一個克蘇魯神話 TRPG 的怪物招式設計師。
請為以下怪物設計完整的招式池。

## 怪物資訊
（同上）

## 設計要求
- 生成 {suggested_count} 張招式卡
- 防禦屬性分佈遵循家族傾向：{tendency}
- 不同招式應涵蓋不同的戰鬥場景（正面衝撞、偷襲、遠程、特殊能力等）
- 至少 1 張招式附帶負面狀態效果
- 權重設計：常用招式 weight 10–15，特殊招式 weight 5–8

## 輸出格式（嚴格 JSON 陣列）
[
  { 同上格式 },
  ...
]

## 施加的負面狀態參考（此家族常用）
{resolved.inflicted_statuses 展開}
```

## 9.4 Gemini Prompt — 敘事描述生成

```
你是一個克蘇魯神話 TRPG 的怪物敘事撰寫者。
請為以下怪物生成 5 個階段的 HP 狀態描述。

## 怪物資訊
- 名稱：{variant.name_zh}
- 種族：{species.name_zh}
- 家族：{family.name_zh}
- 位階：{tier_name}
- 家族主題：{family.theme_zh}

## 要求
- 玩家看不到怪物的 HP 數字，只看到這些描述文字
- 從完好到死亡，描述要有漸進的敘事張力
- 用詞要有克蘇魯神話的恐怖/詭異氛圍
- 每段描述 1–2 句話
- 描述應體現這隻怪物的種族特徵

## 輸出格式（嚴格 JSON 陣列）
[
  { "hp_threshold": 100, "description_zh": "完好狀態描述" },
  { "hp_threshold": 75,  "description_zh": "輕傷狀態描述" },
  { "hp_threshold": 50,  "description_zh": "重傷狀態描述" },
  { "hp_threshold": 25,  "description_zh": "瀕死狀態描述" },
  { "hp_threshold": 0,   "description_zh": "死亡描述" }
]
```

## 9.5 AI 生成的前端流程

1. 使用者點擊「🤖 AI 生成」按鈕
2. 按鈕變為 loading 狀態（轉圈動畫）
3. 前端組裝 Prompt（從當前怪物數據填入模板）
4. 呼叫後端 `/api/admin/monsters/ai/generate`
5. 後端呼叫 Gemini API，取得 JSON 回應
6. 前端解析回應，填入對應欄位
7. **不自動儲存** — 使用者檢視 AI 生成的結果，滿意後手動儲存

### 後端 AI 端點

| 方法 | 路徑 | 功能 |
|------|------|------|
| POST | `/ai/generate-attack-card` | 生成單張招式卡 |
| POST | `/ai/generate-attack-pool` | 生成整個招式池 |
| POST | `/ai/generate-status-descriptions` | 生成敘事描述 |

請求格式：
```json
{
  "variant_id": "uuid",
  "mode": "single_card | full_pool | status_descriptions"
}
```

後端從資料庫撈取該變體的完整資訊（含種族、家族），組裝 Prompt，
呼叫 Gemini，解析回應後回傳給前端。

## 11.2 刪除確認

所有刪除操作彈出確認對話框：

```

└───────────────────────────────────┘
```

## 11.3 位階切換時的數值提示

管理員在變體編輯中切換位階時：

1. 彈出提示：「是否將數值重置為新位階的建議中間值？」
2. 選擇「是」→ 所有戰鬥數值重置
3. 選擇「否」→ 只更改位階標記，保留原數值
4. 無論選擇何者，數值超出新位階建議範圍時顯示黃色警告邊框

## 11.4 繼承值的視覺區分

整個編輯器遵循一致的繼承值視覺規範：

| 狀態 | 視覺處理 |
|------|---------|
| 自身設定值 | 正常文字色 `--text-primary` + 正常邊框 |
| 繼承自種族 | 淡色文字 `--text-secondary` + 虛線邊框 + tooltip「繼承自種族: {species.name_zh}」 |
| 繼承自家族 | 更淡文字 `--text-tertiary` + 虛線邊框 + tooltip「繼承自家族: {family.name_zh}」 |

設計師一眼就能分辨哪些值是自己設定的、哪些是繼承來的。

# 第十二部分：設計狀態追蹤

## 12.1 三級設計狀態

所有層級（家族、種族、變體）共用同一套設計狀態：

| 狀態 | 代碼 | 含義 |
|------|------|------|
