### 自動建議邏輯

- 輸入行動點花費時，自動建議強度標籤（但設計師可覆寫）
  - 1–2 點 → 建議「小型事件」
  - 3–4 點 → 建議「中型事件」
  - 5–6 點 → 建議「大型事件」
  - 7+ 點 → 建議「史詩事件」
- 當「發動時機」不是「調查員階段響應」時，響應觸發條件欄位隱藏
- 當「卡片類型」設為「響應取消類」時，強制發動時機為「調查員階段響應」

### 各動作的參數編輯表單

每個動作代碼都有對應的參數表單。以下是主要動作的編輯 UI：

**召喚怪物 (summon_monster)：**

| 參數 | UI | 說明 |
|------|-----|------|
| 家族池 | 下拉選單 | 從 `monster_families` 表取所有 `is_active=true` 的家族 |
| 數量 | 數字輸入 | 1–5 |
| 基礎位階 | 下拉選單 | 雜兵/威脅/精英/頭目（巨頭不出現在召喚中） |
| 生成位置規則 | 下拉選單 | 從 TARGET_RULES 中 scope=location 的規則 |

**推進議程 (advance_agenda)：**

| 參數 | UI | 說明 |
|------|-----|------|
| 毀滅標記數量 | 數字輸入 | 1–5 |

**環境改變 (environment_change)：**

| 參數 | UI | 說明 |
|------|-----|------|
| 改變類型 | 下拉選單 | darkness / fire / disconnect / haunting |
| 目標地點規則 | 下拉選單 | 從 TARGET_RULES 中 scope=location 的規則 |

**施加狀態 (inflict_status)：**

| 參數 | UI | 說明 |
|------|-----|------|
| 狀態代碼 | 下拉選單 | 從 NEGATIVE_STATUSES 中選 |
| 數值 | 數字輸入 | 通常 1–3 |
| 目標規則 | 下拉選單 | 從 TARGET_RULES 中 scope=investigator 的規則 |

**全場傷害 (damage_all)：**

| 參數 | UI | 說明 |
|------|-----|------|
| 物理傷害 | 數字輸入 | 0–5 |
| 恐懼傷害 | 數字輸入 | 0–5 |
| 目標規則 | 下拉選單 | 通常是 all_investigators |

**混沌袋操作 (modify_chaos_bag)：**

| 參數 | UI | 說明 |
|------|-----|------|
| 操作類型 | 下拉選單 | add（加入） / remove（移除） / replace（替換） |
| 標記類型 | 下拉選單 | 所有混沌袋標記類型（數字、情境、神話、動態） |
| 數量 | 數字輸入 | 1–5 |

**強制重擲 (force_reroll)：**

| 參數 | UI | 說明 |
|------|-----|------|
| 目標規則 | 下拉選單 | last_attacker / lowest_hp / 等 |
| 使用較差結果 | 開關 | 是/否 |

> 完整清單見 Part 1 §1.8 `MYTHOS_ACTION_CODES`。
> Claude Code 建置時需為每個動作代碼實作對應的參數表單元件。

### 自動文字描述生成

每個動作編輯後，系統根據 `action_code` 和 `action_params` 自動生成人類可讀的文字描述，顯示在動作卡片上。

自動生成邏輯範例：

```javascript
function generateActionDescription(actionCode, params) {
  switch (actionCode) {
    case 'summon_monster':
      const family = getFamilyName(params.family_code);
      const tier = getTierName(params.base_tier);
      const location = getLocationRuleName(params.location_rule);
      return `從${family}池中召喚 ${params.quantity} 隻${tier}級怪物於${location}`;

    case 'advance_agenda':
      return `議程牌堆放置 ${params.doom_tokens} 個毀滅標記`;

    case 'inflict_status':
      const status = getStatusName(params.status_code);
      const target = getTargetRuleName(params.target_rule);
      return `對${target}施加 ${params.value} 點${status}狀態`;

    // ...其他動作代碼
  }
}
```

設計師可覆寫自動生成的文字（若需要更精確的敘事）。

一張神話卡可包含多個動作。動作列表可拖曳排序，執行時依 `sort_order` 順序觸發。

### 選項編輯邏輯

- **「需要檢定」開關**：
  - 開：顯示檢定屬性 + DC 欄位 + 成功/失敗兩個結果區塊
  - 關：只顯示一個「結果」區塊（無條件執行）
- **選項數量**：最少 2 個，最多 3 個。刪除按鈕在只有 2 個時禁用，新增按鈕在已有 3 個時禁用。
- **選項標籤**：自動分配 A、B、C，不需手動設定

### 選項編輯的特殊規則

1. **至少一個選項必須需要檢定**：若所有選項都不需檢定，提示設計師「遭遇卡建議至少有一個需要檢定的選項，否則玩家選擇缺乏張力」（警告級，不阻止儲存）

# 第七部分：儲存邏輯

## 7.1 神話卡儲存

- 基本資訊 + 機制屬性 + 視覺 + 設計備註 → 統一 PUT `/mythos-cards/:id`
- 動作列表 → 個別 CRUD（新增、編輯、刪除、排序都是獨立 API）

## 7.2 遭遇卡儲存

- 基本資訊 + 情境描述 + 視覺 + 設計備註 → 統一 PUT `/encounter-cards/:id`
- 地點風格標籤 → PUT `/encounter-cards/:id/tags`（整組覆寫）
- 選項 → 個別 CRUD（每個選項獨立儲存）
- 效果清單 → 儲存在選項的 JSONB 欄位，跟著選項一起儲存

## 7.3 未儲存變更提示

- 有未儲存變更時，右下角浮動儲存按鈕亮起金色光暈
- 切換到其他卡片時，若有未儲存變更，彈出確認對話框：
  - 「儲存並切換」
  - 「放棄變更並切換」
  - 「取消」

# 第九部分：設計原則提醒

## 9.1 神話卡是城主的武器，玩家看不到內容

- 神話卡的文字內容（名稱、描述、動作參數）**對玩家完全隱藏**
- 玩家只會看到城主「打出神話卡」的結果（召喚、傷害、環境改變等）和風味文字（城主朗讀）
- 因此設計時重點在「機制準確」而非「玩家閱讀友善」

## 9.2 遭遇卡是玩家看得到的敘事

- 遭遇卡的情境描述和選項文字**玩家都看得到**
- 設計時重點在「敘事清晰、選擇有張力、後果有故事感」
- 「故事的後半段」（成功/失敗敘事）是讓玩家感受選擇重量的關鍵

## 9.3 動作代碼是固定枚舉，不能隨意新增

- 神話卡的 `action_code` 必須從預定義清單中選擇（見 Part 1 §1.8）
- 遊戲引擎根據 `action_code` 執行對應邏輯，未定義的代碼會被忽略
- 若需要新的動作類型，應通過管理員討論後統一新增到 `MYTHOS_ACTION_CODES` 常數中
