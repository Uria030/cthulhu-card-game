## 一、Gemini AI 敘事生成整合

### 1.1 設計原則

依 Q3 的決議，Gemini 只介入**文字層面**，不碰機制或數值：

| AI 生成 | 範圍 |
|---------|------|
| ✅ 會做 | 背景故事、能力文字、傳敘（引文）、弱點敘事 |
| ❌ 不做 | 屬性配點、簽名卡機制效果、牌組組合、戰鬥熟練選擇 |

### 1.2 生成模式總覽

設計器提供 **4 種 AI 生成模式**，每一種都在不同介面位置觸發：

| 模式 | 觸發位置 | 生成內容 |
|------|---------|---------|
| 模式 A：調查員背景生成 | 分頁 ① 基本資訊的「✨ AI 生成敘事」 | 姓名 + 稱號 + 背景故事 + 能力文字 |
| 模式 B：簽名卡傳敘生成 | 簽名卡設計器的「✨ AI 生成傳敘」 | 傳敘文字（卡面引文） |
| 模式 C：弱點敘事生成 | 個人弱點編輯器的「✨ AI 生成背景」 | 背景故事 + 傳敘 |
| 模式 D：批次補完 | 總覽面板的「批次 AI 補完」 | 對多個尚未完成的模板一次生成 |

### 1.3 前端 API 整合方式

使用 Gemini API。前端透過 `POST /api/admin/gemini/generate-investigator-narrative` 呼叫，後端轉發並附上系統 Prompt。

```javascript
// 前端呼叫範例
async function generateInvestigatorNarrative(templateId, mode, params = {}) {
  const response = await fetch('/api/admin/gemini/generate-investigator-narrative', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      template_id: templateId,
      mode: mode,  // 'background' | 'signature_flavor' | 'weakness_narrative' | 'batch'
      params: params
    })
  });
  return await response.json();
}
```

### 1.4 模式 A：調查員背景生成

**觸發時機：** 設計者在分頁 ① 已填入 MBTI、職業序號、偏重字母、時代標籤後，點擊「✨ AI 生成敘事」。

**輸入參數（params）：**
```json
{
  "mbti_code": "ENTJ",
  "career_index": 2,
  "dominant_letter": "N",
  "dominant_faction": "天啟",
  "era_tags": "1925, 紐約, 通靈者",
  "user_hint": "我希望這是一位女性角色，戰爭寡婦"
}
```

**系統 Prompt 範本：**

```
你是克蘇魯神話冒險遊戲的角色設計師，專長是 1920–30 年代美國背景的偵探與調查員角色。

請為一位調查員設計完整的背景設定，產出以下內容：

1. 姓名（中英文）：符合 1920–30 年代美國社會的姓名。若使用者提供特殊時代標籤（如古代、跨時代），姓名可依該時代調整。
2. 稱號：一個簡短、具辨識度的職業/身份稱號（如「戰地醫師」「唐人街偵探」「私人圖書館員」）。
3. 背景故事（150–250 字中文）：包含角色的出身、目前處境、為何捲入克蘇魯神話事件。必須符合暗黑哥德的氛圍，避免輕鬆戲謔的調性。
4. 能力文字（1–2 句中文）：以非機制語言描述此角色的核心能力傾向。不要出現數值、機率、擲骰等術語。

角色的 MBTI 人格類型與陣營背景：
- MBTI: {mbti_code}（{mbti_zh_name}）
- 職業偏重：{dominant_faction} 陣營
- 偏重陣營的核心態度：{faction_attitude_description}
- 偏重陣營的機制關鍵字：{faction_mechanics}

時代與情境標籤：{era_tags}
使用者額外提示：{user_hint}

設計約束：
- 避免使用中國大陸用語，所有敘事採台灣繁體中文
- 克蘇魯專有名詞採台灣 TRPG 社群慣用譯名（如「克蘇魯」「奈亞拉托提普」「阿卡姆」）
- 不可包含輕鬆幽默的口吻
- 不可描述角色「戰勝」任何舊日支配者或外神
- 敘事留白比填滿更好，讓玩家有想像空間

輸出格式（JSON）：
{
  "name_zh": "...",
  "name_en": "...",
  "title_zh": "...",
  "title_en": "...",
  "backstory": "...",
  "ability_text_zh": "..."
}
```

**預期輸出範例：**

```json
{
  "name_zh": "艾琳娜·霍華德",
  "name_en": "Elena Howard",
  "title_zh": "戰地殘夢者",
  "title_en": "The Battlefield Dreamer",
  "backstory": "艾琳娜在 1917 年成為戰地護士，在馬恩河的壕溝裡抱著垂死的士兵時，她第一次聽見了『那些聲音』。戰爭結束後她回到紐約，試圖用白色醫院的消毒水味蓋掉噩夢，卻發現那些聲音沒有停止。現在她在哈林區經營一間小診所，白天照顧窮人，夜晚記錄著夢中聽到的字詞——她知道那不是英語，也不是任何活著的人會說的語言。最近她診所的病人開始做一樣的夢。",
  "ability_text_zh": "艾琳娜的夢境與現實交疊。她能在混沌中辨識出將臨的模式，但每一次準確的預感都會削弱她對現實的掌握。"
}
```

### 1.5 模式 B：簽名卡傳敘生成

**觸發時機：** 在簽名卡設計器中，設計者已填入卡名、類型、效果後，點擊「✨ AI 生成傳敘」。

**輸入參數：**
```json
{
  "investigator_id": "uuid-of-investigator",
  "card_name_zh": "最後一發",
  "card_type": "event",
  "card_effect_text": "擲 d20 攻擊一次，必定命中。本場戰鬥中不能再使用任何槍枝類卡片。",
  "investigator_context": {
    "name_zh": "艾琳娜·霍華德",
    "title_zh": "戰地殘夢者",
    "backstory_summary": "1917 戰地護士..."
  }
}
```

**系統 Prompt 範本：**

```
你是克蘇魯神話卡牌遊戲的敘事設計師。

請為一張簽名卡撰寫「傳敘」（Flavor Text）。傳敘是印在卡面底部的引文或敘事片段，長度 20–60 字中文，作用是營造氛圍而非說明機制。

簽名卡資訊：
- 卡名：{card_name_zh}
- 類型：{card_type_zh}
- 機制效果：{card_effect_text}

歸屬調查員：
- 姓名：{investigator_name_zh}
- 稱號：{investigator_title_zh}
- 背景摘要：{backstory_summary}

傳敘設計原則：
1. 用第一人稱內心獨白、對話片段、或第三人稱旁白其中一種。
2. 不可描述機制（不要說「造成 X 點傷害」「擲 d20」等）。
3. 呼應卡片的機制精神，但用氛圍語言表達。
4. 克蘇魯基調：不安、孤獨、無力感、未知、禁忌的誘惑。
5. 避免輕鬆戲謔。

範例風格：
- 「她數了數彈匣裡的子彈。六發。外面有七個。」
- 「『主啊，』他低語，『祢真的還在聽嗎？』」
- 「書頁翻開的聲音，像某個非常古老的東西在笑。」

輸出格式（JSON）：
{
  "flavor_text": "..."
}
```

### 1.6 模式 C：弱點敘事生成

**觸發時機：** 在個人弱點編輯器中，設計者已填入弱點名稱、類型、觸發條件、負面效果後，點擊「✨ AI 生成背景」。

**輸入參數：**
```json
{
  "investigator_id": "uuid",
  "weakness_name_zh": "舊傷復發",
  "weakness_type": "trauma",
  "trigger_condition": "抽到此卡時",
  "negative_effect": "本回合損失 2 點 HP，且力量檢定 -2",
  "investigator_context": { ... }
}
```

**系統 Prompt 範本：**

```
你是克蘇魯神話冒險遊戲的敘事設計師。

請為一張個人弱點卡撰寫背景故事與傳敘。個人弱點是調查員內心或身體的缺陷，強制納入牌組，抽到時觸發負面效果。

弱點資訊：
- 名稱：{weakness_name_zh}
- 類型：{weakness_type}（{type_description}）
- 觸發條件：{trigger_condition}
- 負面效果：{negative_effect}

歸屬調查員：
- 姓名：{investigator_name_zh}
- 稱號：{investigator_title_zh}
- 背景：{backstory}

請產出：
1. 背景故事（80–150 字中文）：解釋這個弱點的由來，為什麼這個調查員會帶著這個缺陷行動。
2. 傳敘（20–50 字中文）：一段氛圍式的描寫。

設計原則：
- 弱點不是單純的懲罰，是角色真實的「傷」。
- 不要描述角色如何「克服」弱點，弱點的存在本身就是克蘇魯的一部分。
- 保持沉重、內省的語氣。

輸出格式（JSON）：
{
  "backstory": "...",
  "flavor_text": "..."
}
```

### 1.7 模式 D：批次補完

**觸發時機：** 總覽面板的「批次 AI 補完」按鈕。

**選擇邏輯：** 
- 設計者先在矩陣中選擇要補完的模板（多選勾選）
- 限制單次批次最多 8 個模板（避免 token 爆量與等待過久）
- 每個模板獨立呼叫 Gemini（並行），不合併 prompt

**前端互動：**


### 1.8 Gemini API 呼叫的技術細節

**後端路由：**

```javascript
// /api/admin/gemini/generate-investigator-narrative
router.post('/generate-investigator-narrative', async (req, res) => {
  const { template_id, mode, params } = req.body;

  try {
    // 1. 取出調查員當前資料
    const investigator = await db.getInvestigator(template_id);

    // 2. 根據 mode 組裝 Prompt
    const systemPrompt = buildSystemPrompt(mode, investigator, params);

    // 3. 呼叫 Gemini
    const response = await gemini.generateContent({
      model: 'gemini-pro',
      contents: [{ role: 'user', parts: [{ text: systemPrompt }] }],
      generationConfig: {
        temperature: 0.85,  // 較高溫度鼓勵創意
        maxOutputTokens: 800,
        responseMimeType: 'application/json'
      }
    });

    // 4. 解析 JSON 回應
    const result = JSON.parse(response.text());

    // 5. 回傳（不自動寫入 DB，讓設計者手動確認）
    res.json({ success: true, data: result });
  } catch (err) {
    res.status(500).json({ success: false, error: err.message });
  }
});
```

**重要：** AI 生成的結果**不自動寫入 DB**。設計者在前端看到結果後，決定是否採用、部分採用、或要求重新生成。這是「理想敘事生成」而非「自動填寫」的核心分界。

## 二、總覽面板

### 2.1 定位

總覽面板是 MOD-11 的第二個主要介面（第一個是 16×4 矩陣編輯器）。總覽面板是**唯讀的分析視圖**，讓設計者了解整個 64 模板庫的設計進度與分佈。

### 2.2 進入方式

在 MOD-11 主頁面的頂部導航列增加「📊 總覽面板」按鈕，點擊切換到總覽視圖。

### 2.3 總覽面板佈局

總覽面板採用**四宮格佈局**：

```
┌─────────────────────────────────────────────────────────┐
│  📊 調查員模板總覽                                       │
├────────────────────────────┬────────────────────────────┤
│                            │                            │
│   面板 A：完成度矩陣        │    面板 B：完成度圖表        │
│   （16×4 視覺化狀態圖）     │    （圓餅圖 + 柱狀圖）       │
│                            │                            │
├────────────────────────────┼────────────────────────────┤
│                            │                            │
│   面板 C：時代標籤分佈      │    面板 D：偏重字母分佈      │
│   （標籤雲 + 出現次數）     │    （八陣營柱狀圖）          │
│                            │                            │
└────────────────────────────┴────────────────────────────┘
```

### 2.4 面板 A：完成度矩陣

以熱力圖（heatmap）形式呈現 16×4 矩陣：

```
          職業 1   職業 2   職業 3   職業 4
INTJ     [█]     [█]     [░]     [░]
INTP     [█]     [░]     [░]     [░]
ENTJ     [█]     [█]     [█]     [░]
ENTP     [░]     [░]     [░]     [░]
...
```

**顏色編碼：**
- `[█]` 深金色：已完成（`is_completed = TRUE`）
- `[▓]` 中灰色：進行中（有部分欄位填寫但未達完成）
- `[░]` 深灰色：骨架（尚未開始）

**互動：** 點擊任一格直接跳轉到該模板的編輯介面。

### 2.5 面板 B：完成度圖表

**圓餅圖：** 完成 / 進行中 / 骨架 的比例。

```
╔══════════════════════╗
║  總體完成度           ║
║                      ║
║    ████ 已完成 19%    ║
║    ▓▓▓▓ 進行中 31%   ║
║    ░░░░ 骨架    50%   ║
║                      ║
║  12 / 64             ║
╚══════════════════════╝
```

**柱狀圖：** 各 MBTI 群組的完成比例（4 群組 × 16 個模板）。

### 2.6 面板 C：時代標籤分佈

所有模板的 `era_tags` 欄位去重後統計。以**標籤雲**呈現，出現次數越多字體越大：

```
    1925   紐約   阿卡姆
  1920s                 偵探
         大學教授    考古隊
    私家偵探    通靈者   軍人
              1928
```

每個標籤旁以小字顯示出現次數，如「1925 (8)」。

### 2.7 面板 D：偏重字母分佈

統計 64 個模板中 `dominant_letter` 的分佈。預設狀態下應該是**均勻分佈**（每個字母 8 個模板），但若設計者未來新增自建模板可能會偏差。

柱狀圖顯示：

```

I ████████ 8
S ████████ 8
N ████████ 8
T ████████ 8
F ████████ 8
J ████████ 8
P ████████ 8
```

若某個字母偏多或偏少，顯示警告：「⚠ I 偏重模板過多（12 個），建議平衡設計」。

## 三、與其他模組的整合點

### 3.1 與 MOD-05 戰鬥風格的整合

**讀取：** 分頁 ③ 戰鬥熟練下拉選單讀取 `proficiency_definitions` 表所有項目。

**無寫入：** MOD-11 不修改 MOD-05 的資料。

### 3.2 與 MOD-01 卡片設計器的整合

**讀取：** 分頁 ⑤ 起始牌組的「可用卡池」讀取 `card_definitions` 表，**依該調查員的 MBTI 四字碼過濾**。

**過濾邏輯：**
```sql
SELECT * FROM card_definitions
WHERE faction_code IN ('E', 'N', 'T', 'J', NULL, 'neutral')  -- ENTJ 的可用陣營
  AND card_type != 'signature'  -- 排除簽名卡
  AND card_type != 'weakness'   -- 排除弱點
ORDER BY cost ASC, name_zh ASC;
```

**無寫入：** MOD-11 不修改 MOD-01 的卡池資料。

### 3.3 與 MOD-10 城主設計器的關聯

**無直接整合。** 城主的遭遇卡若引用「調查員屬性」是透過結構化動作代碼（如 `force_check_all` + 屬性參數），不需要 MOD-11 提供具體調查員資料。

### 3.4 與玩家端創角器的關聯

**最終目標：** 64 個預設模板會供玩家創角器讀取。

## 六、實作注意事項彙整

### 6.1 敘事欄位的「留白哲學」

MOD-11 最大的特色是**敘事留白**。所有預設模板的 64 個格子都「預先存在但空白」，等待設計者填寫。這設計意圖：

1. **讓設計者感受到工作量與結構**：打開設計器就看到 64 個待填的格子，心理負擔透明化
2. **避免 AI 自動填滿的鄉愿**：AI 只是輔助，最終決定權在人類設計者
3. **保留不同設計者風格的可能性**：未來可能有多個設計師分工填寫不同區塊

實作時請嚴格遵守這個哲學：
- 預設模板骨架**絕對不可**預填任何敘事內容
- AI 生成結果**絕對不可**自動寫入 DB
- 所有填寫都必須經過設計者在介面上的「採用」動作

### 6.2 MBTI 矩陣的 UX 細節

- 矩陣載入時按 MBTI 群組（分析家/外交家/守護者/探險家）分組顯示
- 每個群組標題可摺疊，方便設計者專注於某群組
- 已完成的模板用金色邊框，視覺上快速識別進度
- 鼠標懸停顯示完整資訊（姓名、稱號、屬性摘要）

### 6.3 起始牌組構築的常見陷阱

實作時注意：
- 陣營過濾必須處理「中立卡」的邊界情況（`faction_code IS NULL` 或字串 'neutral'）
- 簽名卡和弱點加入牌組後不可透過「-」按鈕移除（需從簽名卡/弱點編輯器端刪除）
- 數量調整有上限（例：大部分卡片最多 2 張，傳奇卡最多 1 張，此規則由 MOD-01 卡片定義決定）

### 6.4 Gemini API 配額與錯誤處理

- 批次補完應限制同時並行數量（建議最多 4 個並行請求）
- 單次呼叫失敗時重試最多 2 次，否則放棄該模板並記錄
- 若 API 回傳非 JSON 格式（Gemini 偶發異常），嘗試解析失敗時顯示錯誤並讓設計者重試
- 所有 AI 生成的原始 Prompt 與回應應寫入日誌（供品質檢討）
