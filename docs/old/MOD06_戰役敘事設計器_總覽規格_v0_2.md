## 一、模組定位

### 1.1 本模組要解決的問題

戰役敘事設計器是**整部戰役的劇情骨架與結構容器**。它不負責任何可玩單元的細節——那是關卡編輯器(MOD-07)的工作。它負責的是:

- 把十個章節排成序列
- 定義每一章的**敘事起點**(章首事件)與**結算尾聲**(章末事件)
- 定義每一章的**結果分支**(至少 2 種、至多 5 種結果)與**混沌袋演變規則**
- 統一管理**旗標字典**——戰役內所有旗標的命名、分類、來源
- 管理**戰役元資料**——名稱、主題、封面敘事、起始混沌袋配置

### 1.2 邊界

| 是 MOD-06 的職責 | 不是 MOD-06 的職責 |
|---|---|
| 章節元資料、序列、分支走向 | 關卡本體結構(交給 MOD-07) |
| 章首／章末事件的觸發條件與執行內容 | 目標牌堆、議案牌堆本身(交給 MOD-07) |
| 結果判定規則(進度與旗標組合→結果代號) | 戰役進行中實際結算結果的邏輯(執行期) |
| 混沌袋演變規則(結果→標記變化) | 關卡現場的初始混沌袋組成(交給 MOD-07) |
| 旗標字典(命名、分類、描述) | 旗標的執行期值(執行期 Campaign Log) |
| — | 個人秘密任務(獨立於 MOD-13) |

### 1.3 與其他模組的對接

- **MOD-07 關卡編輯器**:MOD-06 提供章節骨架;MOD-07 產出關卡時引用所屬章節代碼,通關授予的旗標必須在 MOD-06 旗標字典中存在
- **MOD-10 城主設計器**:章首／章末事件若包含城主敘事介入,引用城主模板
- **MOD-03 敵人設計器**:章末事件可執行操作中包含生成敵人,引用怪物家族代碼
- **MOD-04 團隊精神**:章末結算可能授予凝聚力,引用團隊精神代碼(例如解鎖怪物學家族)
- **MOD-13 秘密任務編輯器**(未來):個人秘密任務由 MOD-13 獨立管理,與 MOD-06 無直接資料依賴

## 二、功能範圍

### 2.1 戰役管理

- 建立／編輯／刪除戰役
- 戰役元資料編輯:名稱(中英)、主題、封面敘事、設計狀態、版本
- 戰役起始混沌袋配置(數字標記、情境標記、神話標記的起始組成,含每類標記的數值)
- 戰役難度基準(簡單／標準／困難／專家)
- **建立戰役時自動產生十章空骨架**(編號 1–10、預設章節縮寫 `ch1`–`ch10`、設計狀態 draft)
- 戰役章節序列總覽(十章骨架,顯示每章狀態)

### 2.2 章節編輯

每一章包含以下可編輯元素:

- **基礎資料**:章節編號(1–10,不可變)、名稱(中英)、章節縮寫(用於旗標命名前綴,預設 `ch1`–`ch10`,可自訂)、敘事定位
- **劇情演示**:進入章節時的劇情文字與劇情選項(影響分支走向)
- **關卡掛載清單**:本章包含的關卡清單(僅顯示引用,實際編輯在 MOD-07)
- **章首事件**(prologue):觸發條件、執行操作、敘事文字
- **章末事件**(epilogue):觸發條件、執行操作、敘事文字
- **結果分支定義**:2–5 種結果代號(**限定為 A/B/C/D/E 五個槽位**),每種結果的判定條件(由目標進度、議案進度、旗標組合表達)
- **結果導向**:每種結果指向下一章的哪個版本
- **混沌袋演變規則**:每種結果帶來的永久混沌袋變化(例如 C 結果新增邪教徒標記 ×2)
- **章節完成度**:自動計算(所有必填欄位齊備且至少設計 1 個關卡且至少 2 種結果分支)

### 2.3 旗標字典

戰役內所有旗標的集中管理介面。旗標不在關卡編輯器裡憑空產生,必須先在旗標字典登記,才能被任何模組引用。

- 旗標命名規則校驗(`[類別].[章節縮寫]_[描述]`)
- 旗標九大類別(行動進度 / 議程進度 / NPC 狀態 / 物品取得 / 地點探索 / 玩家選擇 / 章節結果 / 時間狀態 / 隱藏旗標)
- 旗標可見性設定(可見 / 視設計 / 永遠隱藏)
- 旗標來源追蹤(自動反向查詢:哪些章節產生、哪些章節／關卡讀取)
- 旗標描述(提醒設計師這個旗標代表什麼)
- 支線旗標警示(若旗標被標記為「支線專屬」,其他章節不應依賴)

### 2.4 章首／章末事件編輯器

間章事件屬於戰役劇情的一部分,**每個事件綁定單一章節的單一插入點**(不跨章節複用,因戰役為完整小說劇情拆解)。編輯內容:

- 事件名稱
- 插入點(`prologue` 或 `epilogue`)
- 綁定章節(單一)
- 觸發條件(旗標組合表達式;空值 = 無條件觸發)
- 執行操作清單(支援六類操作:損耗物資、設定旗標、觸發檢定、給予選擇、授予獎勵、施加懲罰)
- 敘事文字(中英)
- 分支結果(若事件含選擇,每個選項的導向)

### 2.5 混沌袋演變規則

每個戰役的混沌袋從起始組成開始,**隨章節推進逐漸變差**。此介面管理的是「結果→標記變化」的規則。

- 每章的每種結果各自定義一組演變規則
- 演變規則由一組**只記變化**的操作構成:加入某類標記 X 顆 / 移除某類標記 X 顆 / 替換標記
- 標記的**數值**由戰役起始混沌袋配置決定(例如起始時邪教徒已定義為 -2),演變只管進出,不重新定義數值
- 標記類別完整支援:數字標記、情境標記(骷髏/邪教徒/石版/遠古邪物)、神話標記(線索/頭條/怪物/毀滅/次元門)、極端標記、動態標記(祝福/詛咒)
- 與戰役起始混沌袋聯動的**預計終局混沌袋**模擬器(依走最壞路徑、最好路徑、平均路徑)

### 2.6 完整性檢查

戰役發佈前的自動檢查器。產出一份清單告訴設計師還缺什麼。

## 三、資料結構概觀

> 以下列出主要實體與欄位,供後續資料庫遷移指令文件展開。欄位名稱採用既有命名慣例(snake_case)與現有表一致。

### 3.1 `campaigns`(戰役)

| 欄位 | 類型 | 說明 |
|---|---|---|
| `id` | UUID | 主鍵 |
| `code` | VARCHAR(32) | 戰役代碼(唯一) |
| `name_zh` / `name_en` | VARCHAR(128) | 名稱 |
| `theme` | VARCHAR(64) | 主題(例:印斯茅斯、敦威治等) |
| `cover_narrative` | TEXT | 封面敘事 |
| `difficulty_tier` | VARCHAR(16) | 難度基準(easy/standard/hard/expert) |
| `initial_chaos_bag` | JSONB | 起始混沌袋配置(含每類標記的數值定義) |
| `design_status` | VARCHAR(16) | 設計狀態(draft/review/published) |
| `version` | INTEGER | 版本 |

> **後端補充**:`POST /api/campaigns` 在交易內自動建立十筆 `chapters` 紀錄,失敗則整筆戰役建立回滾。

### 3.2 `chapters`(章節)

| 欄位 | 類型 | 說明 |
|---|---|---|
| `id` | UUID | 主鍵 |
| `campaign_id` | UUID | 外鍵(ON DELETE CASCADE) |
| `chapter_number` | INTEGER | 1–10(CHECK 約束) |
| `chapter_code` | VARCHAR(16) | 縮寫(如 `ch3`),旗標命名用 |
| `name_zh` / `name_en` | VARCHAR(128) | 名稱 |
| `narrative_intro` | TEXT | 劇情演示文字 |
| `narrative_choices` | JSONB | 劇情演示選項 |
| `design_status` | VARCHAR(16) | 設計狀態 |

唯一鍵:`(campaign_id, chapter_number)`、`(campaign_id, chapter_code)`

### 3.3 `chapter_outcomes`(章節結果分支)

每章 2–5 筆。

| 欄位 | 類型 | 說明 |
|---|---|---|
| `id` | UUID | 主鍵 |
| `chapter_id` | UUID | 外鍵(ON DELETE CASCADE) |
| `outcome_code` | VARCHAR(1) | **A/B/C/D/E(CHECK 約束,固定五槽)** |
| `condition_expression` | JSONB | 判定條件(旗標組合) |
| `narrative_text` | TEXT | 結果敘事 |
| `next_chapter_version` | VARCHAR(16) | 下一章的版本指向(終局章節為 NULL) |
| `chaos_bag_changes` | JSONB | 混沌袋演變規則(只記操作) |
| `rewards` | JSONB | 結算獎勵(經驗值、天賦點、凝聚力等) |
| `flag_sets` | JSONB | 此結果設定的旗標清單 |

唯一鍵:`(chapter_id, outcome_code)`

### 3.4 `campaign_flags`(旗標字典)

| 欄位 | 類型 | 說明 |
|---|---|---|
| `id` | UUID | 主鍵 |
| `campaign_id` | UUID | 外鍵(ON DELETE CASCADE) |
| `flag_code` | VARCHAR(128) | 完整旗標代碼(含類別前綴) |
| `category` | VARCHAR(16) | 九大類別之一 |
| `description_zh` | TEXT | 描述 |
| `visibility` | VARCHAR(16) | visible/conditional/hidden |
| `chapter_code` | VARCHAR(16) | 所屬章節(若為章節範疇) |

唯一鍵:`(campaign_id, flag_code)`

### 3.5 `interlude_events`(間章事件)

| 欄位 | 類型 | 說明 |
|---|---|---|
| `id` | UUID | 主鍵 |
| `chapter_id` | UUID | **單一綁定章節(外鍵,ON DELETE CASCADE)** |
| `event_code` | VARCHAR(64) | 事件代碼 |
| `name_zh` / `name_en` | VARCHAR(128) | 名稱 |
| `insertion_point` | VARCHAR(16) | `prologue` / `epilogue` |
| `trigger_condition` | JSONB | 觸發條件(null = 無條件) |
| `operations` | JSONB | 執行操作清單 |
| `narrative_text_zh` | TEXT | 中文敘事 |
| `narrative_text_en` | TEXT | 英文敘事 |
| `choices` | JSONB | 選項與導向 |

> **註**:同一章節同一插入點可有多個事件,依觸發條件決定是否觸發;全部無觸發條件則依建立順序全部執行。

### 3.6 引用型關聯(不建新表)

- **關卡引用**:MOD-07 的 `stages` 表會有 `chapter_id` 外鍵。MOD-06 的章節編輯器只做反向查詢顯示。
- **城主引用**:間章事件的 `operations` 若需城主介入,以代碼引用 MOD-10 的城主模板。
- **怪物家族引用**:間章事件的 `operations` 若需生成敵人,以代碼引用 MOD-03 的家族代碼。

## 四、介面結構

### 4.1 整體佈局

沿用 `admin-shared.css` 暗黑哥德風格,三欄式佈局:

- **左側欄**:戰役列表 + 當前戰役的章節序列樹狀圖
- **主編輯區**:依當前分頁顯示對應編輯器
- **右側欄**:即時預覽(章節卡、事件卡、結果分支流程圖)

### 4.2 分頁結構

| 分頁 | 主編輯區內容 |
|---|---|
| 戰役總覽 | 戰役元資料、起始混沌袋、章節骨架十格顯示 |
| 章節編輯 | 選定章節的所有欄位(基礎、劇情演示、結果分支、關卡引用清單) |
| 旗標字典 | 戰役內旗標清單、新增、反向引用追蹤 |
| 間章事件 | 事件清單(依章節分組)、編輯器、觸發條件 |
| 混沌袋演變 | 每章每結果的演變規則 + 終局模擬器 |
| 完整性檢查 | 戰役發佈前檢查清單 |

### 4.3 關鍵操作流程

**建立新戰役**
1. 填寫元資料 → 系統**在同一筆交易內**自動產生十章空骨架(狀態均為 draft、縮寫 `ch1`–`ch10`)
2. 設定起始混沌袋
3. 進入第一章開始編輯

**章節結果分支的編寫流程**
1. 在章節編輯分頁選擇結果數量(2–5,對應 A–E 槽位)
2. 為每個結果寫判定條件(系統提供表達式編輯器,顯示可用旗標下拉)
3. 為每個結果寫敘事文字與混沌袋演變規則
4. 指定下一章的版本指向(終局章節設為 NULL)
5. 指定此結果授予的旗標(系統自動校驗旗標是否在字典中存在)

**間章事件的建立流程**
1. 選擇綁定章節(單一)
2. 選擇插入點(章首或章末)
3. 設定觸發條件(旗標組合;留空代表無條件觸發)
4. 新增執行操作(支援六類操作選單,每類有動態欄位)
5. 撰寫敘事文字
6. 若含選擇,設定每選項導向

### 4.4 條件表達式編輯器

本模組內部共用元件。用於:結果判定條件、間章事件觸發條件。

支援的運算子:
- 邏輯:AND / OR / NOT
- 比較:旗標存在 / 旗標值等於 / 旗標值大於等於
- 進度比較:目標牌堆進度 ≥ X / 議案牌堆進度 ≥ X

JSON 結構範例:
```json
{
  "type": "and",
  "conditions": [
    { "type": "flag_set", "flag_code": "npc.ch5_henry_alive" },
    { "type": "agenda_progress_gte", "value": 2 }
  ]
}
```

### 4.5 混沌袋演變規則 JSON 結構

**只記變化**,不重新定義標記數值:

```json
[
  { "op": "add", "marker": "cultist", "count": 2 },
  { "op": "remove", "marker": "tablet", "count": 1 }
]
```

標記代碼採用規則書第二章 §5.2 的分類代碼(`skull` / `cultist` / `tablet` / `elder_thing` / `bless` / `curse` / `clue` / `headline` / `monster` / `doom` / `gate` 等)。

## 五、AI 整合點

全部透過 Gemini 2.5 Flash(沿用 MOD-01/02/03/08/10 的模式,前端直接呼叫)。

| 生成點 | 輸入上下文 | 輸出格式 |
|---|---|---|
| 戰役封面敘事 | 主題、難度基準、章節序列摘要 | 300–500 字中文敘事 + 英文翻譯 |
| 章節劇情演示 | 戰役主題、本章定位、前一章結果、劇情選項骨架 | 演示文字 + 2–3 個選項 |
| 章末結果敘事 | 本章劇情演示、結果類型(A/B/C 含義)、影響範圍 | 每結果 100–200 字 |
| 間章事件敘事 | 事件觸發條件、執行操作、敘事情境 | 情境描述 + 選項文字 |
| 旗標描述 | 旗標代碼、類別、相關章節 | 1–2 句中文描述 |
| 中譯英 | 任何中文欄位 | 英文翻譯 |

所有 Prompt 模板會嵌入第一章「核心情感設計」的情感目標,確保 AI 產出符合克蘇魯氛圍。

## 六、種子資料要求

開啟模組時預先載入**示範戰役「印斯茅斯陰影」**:

- 戰役元資料完整填寫
- 起始混沌袋配置(參考第六章 §12.3 的「印斯茅斯深淵」範例)
- 十章骨架(章節編號、名稱、縮寫、設計狀態)
- **前兩章完整內容**(供設計師看到「完整的章節長什麼樣」):
  - 第一章:完整劇情演示、章首與章末事件各 1、2 種結果分支、混沌袋演變規則
  - 第二章:同上
- 第一章的旗標字典(約 5–8 個示範旗標,涵蓋多個類別)

種子資料採用台灣 TRPG 社群的克蘇魯神話慣用譯名(依 `克蘇魯神話專有名詞對照表`)。

## 七、完成度檢查規則

### 7.1 章節可發佈的條件

- 基礎資料齊備(名稱、縮寫、敘事定位)
- 劇情演示有文字且至少 1 個選項
- 至少 1 個關卡掛載於本章
- 至少 2 種結果分支
- 每個結果分支有:判定條件、敘事文字、下一章指向(或明確 NULL 表示終局)、混沌袋演變規則
- 所有引用的旗標在字典中存在

### 7.2 戰役可發佈的條件

- 戰役元資料齊備
- 起始混沌袋配置完整(符合第六章 §12 的難度配置)
- 十章均達到「章節可發佈」狀態
- 章與章之間的結果指向無斷裂(每個結果的 `next_chapter_version` 在下一章中存在)
- 第十章為終局章節,所有結果的 `next_chapter_version` 為 NULL
- 旗標字典內無孤立旗標(被定義但從未被引用)
- 旗標字典內無懸空引用(被引用但未定義)

## 八、與其他模組的接口清單

### 8.1 對外提供(供其他模組讀取)

| API 路徑 | 用途 | 主要消費者 |
|---|---|---|
| `GET /api/campaigns` | 戰役列表 | MOD-07 |
| `GET /api/campaigns/:id/chapters` | 章節序列 | MOD-07 |
| `GET /api/campaigns/:id/flags` | 旗標字典 | MOD-07 |
| `GET /api/chapters/:id` | 章節詳情(含結果分支、間章事件) | MOD-07 |

### 8.2 對內接收(其他模組寫入本模組資料)

MOD-06 不接受其他模組直接寫入。若 MOD-07 的關卡通關需要授予旗標,該旗標必須**先**在 MOD-06 旗標字典中登記,MOD-07 在關卡結算規則中僅引用代碼。

### 8.3 校驗關係

| 校驗項 | 校驗邏輯 |
|---|---|
| 旗標引用完整性 | MOD-07 儲存關卡時,後端檢查所授予／讀取的旗標代碼都在 MOD-06 字典中 |
| 章節關卡掛載 | MOD-07 儲存關卡時,`chapter_id` 必須對應 MOD-06 已存在的章節 |
| 下一章指向 | MOD-06 儲存章節結果時,`next_chapter_version` 的目標章節必須存在(終局章節除外) |
