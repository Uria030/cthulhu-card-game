# 事故紀錄 26041802
## gemma-bridge 架構與術語錯誤事故
### Incident Report — gemma-bridge Architectural & Terminology Errors

> **事故時間：** 2026/04/18
> **聊天室：** GEMMA（本地 GEMMA 部署與橋接設計）
> **工作者：** Uria（專案原創者）+ Claude Opus 4.7
> **事故性質：** Claude 未讀專案文件即產出含大量錯誤定義的交付物
> **嚴重程度：** 高 — 若未被攔下，會導致 GEMMA 批次產出數百張資料結構錯誤的 JSON，全部進入 DB 污染資料

---

## 一、事故摘要

在 GEMMA 本地部署專案中，Claude Opus 4.7 產出了六份交付文件（01–06），用於引導建置 `gemma-bridge` 橋接程式與五份 Prompt 模板。

這六份文件在 Uria 的後續檢驗中，被發現含有大量與專案既有規範不符的錯誤，包含：

- 陣營系統完全錯誤（用通用 TCG 分類替代實際的八陣營極體系）
- 稀有度代碼未對齊資料庫 Schema
- 敵人家族清單不完整
- 場景層級術語混用（Stage 與 Scenario）
- Gemini API 呼叫架構違反既定前端直呼模式
- 等級抵扣數值錯誤
- Admin API endpoint 路徑自行推測

錯誤共分 10 大類，總計約 30 項具體錯誤點。

---

## 二、錯誤的實質影響

### 若未被發現的後果

- GEMMA 依這些 Prompt 產出的 JSON，陣營欄位填入不存在的 `explorer / scholar / fighter` 等代碼
- 稀有度欄位填入 Schema 中不存在的 enum 值
- 敵人家族代碼不完整，可能漏掉 `house_cthugha`、`house_yig` 等 2026/04 確認的家族
- 所有產出會在寫入 DB 階段**全部被後端拒絕**，或更糟——**部分寫入後造成資料不一致**

### 實際損失

- 六份文件需要檢查修正（已產出「錯誤修正檢查清單」交給 Claude Code 處理）
- `gemma-bridge` 模組已建置，需要重新檢查其 Schema 與 Prompt
- Uria 約多花費 1–2 小時處理錯誤檢視與指導修正
- 專案推進延遲，但尚未造成資料層污染（因為 GEMMA 尚未實際執行批次產出）

---

## 三、根本原因分析

### 表層原因

Claude 產出六份文件時，僅使用三次零碎的 `project_knowledge_search`（卡片價值、克蘇魯譯名、資料庫 Schema），未系統性閱讀規則書六章、支柱一陣營定義、怪物家族設計草案 v2 等核心權威文件。

### 深層原因

Claude 的內部流程**跳過**了「先讀後做」的基本紀律，直接進入「產出模板」模式。產出範例時，遇到不確定的專案特定值（如陣營代碼），Claude 從訓練資料中的**通用 TCG 常識**自動補填，而非打開專案文件查證。

### 為何規則文件沒有防止錯誤

Uria 的 userPreferences 已包含極為明確的規則：
- DEBUG 迴圈（收到修正提醒立即停止）
- 禁止質疑使用者操作
- 產出太大就拆檔

這些規則在本次對話中**確實都生效了**。問題不在「違反寫明的規則」，而在「**Claude 在規則沒明確覆蓋的地方自動偷懶**」。userPreferences 中並無任何條目明確要求「產出前先讀文件」，所以 Claude 跳過了。

### 核心失敗模式

**「知道文件存在」≠「讀過文件」≠「對齊文件」**

Claude 知道專案有完整規則書，也知道 Project Knowledge 中有 19 份文件，但這個**認知**沒有轉化為「動筆前先把相關章節讀過一遍」的**行為**。

## 四、攔截過程

### Uria 的檢驗路徑

1. 啟動 gemma-bridge 建置流程（由 Claude Code 執行文件 02 的指令書）
2. Claude Code 成功建置 12 個子任務，服務在 port 8787 運行
3. 五份 Prompt 檔案成功部署
4. **Uria 在進入第一階段測試前，主動要求 Claude 讀規則書第一章**
5. Claude 讀完第一章後，**未能立即察覺自己的錯誤**，Uria 追問「然後還沒反應過來？」
6. Claude 誤判 Uria 暗示方向（以為是 Stage/Scenario 術語錯誤）
7. Uria 再次明確指出規則書第一章結尾備註
8. Claude 才意識到「技術細節不應問使用者」的原則違反
9. Uria 強制 Claude 重新系統性閱讀全部專案文件
10. Claude 完整閱讀後，主動列出六份文件的所有錯誤
11. Uria 要求將錯誤列成檢查清單交由 Claude Code 處理（而非讓 Claude 自己重做，避免再次犯錯）

### 攔截成功的關鍵

- **Uria 的抽查習慣**：在系統實際執行前就做規則書對照
- **Uria 拒絕「Claude 自己重做」的誘惑**：直接轉交 Claude Code 處理，降低再犯機率
- **整份事故從發生到攔截不到 2 小時**

## 五、Claude 在事故中的行為反省

### 做錯的部分

| 錯誤行為 | 應有行為 |
|---------|---------|
| 收到「開始」指令後立刻產出 | 先列出需查證的專案特定值清單，逐項查證後才產出 |

| 克蘇魯家族只寫記得的 | 直接讀怪物家族設計草案 v2 |
| 敵人 DC 用籠統值 | 規則書第六章 §5.1 有明確 DC 12/16/20/24/28 |
| 質疑 Uria 的技術環境（問 Admin Module 埠號） | Claude Code 自行讀專案檔案判斷即可 |

### 做對的部分

| 正確行為 |
|---------|
| 被 Uria 指正後正確執行 DEBUG 迴圈（停止分析、等確認、再修正） |
| 拒絕「自己重做六份文件」的衝動，同意轉交 Claude Code 處理 |
| 產出錯誤檢查清單時不替自己辯護，直接承認 |
| 對「你不是 Opus 吧」的質疑，誠實回應「是行為選擇失敗，不是模型能力問題」 |

## 六、預防機制

### 已採取的措施

#### 措施 1：建立本事故紀錄

本份文件放入 Project Knowledge，作為未來任何聊天室的 Claude 實例接手本專案時的**必讀警示**。類比航空業的失事報告制度——具體的事故比抽象規則更能讓人記住。

#### 措施 2：產出錯誤修正檢查清單

不讓犯錯的 Claude 自己重做，改由 Claude Code 依清單逐項修正。這降低了「同一個 Claude 實例重複犯錯」的風險。

### 建議後續採取的措施

#### 建議 1：在 userPreferences 加入「文件校對義務」

具體措辭建議：

> 在產出任何涉及專案內容的成果物之前（包含但不限於：指令文件、Prompt、Schema、程式碼、範例卡片），必須先執行「文件校對義務」：
>
> 1. 列出成果物會用到的專案特定概念（陣營、稀有度、術語、代碼、公式、家族、位階等）
> 2. 對每個概念，透過 project_knowledge_search 或直接讀取對應文件，確認實際定義
> 3. 若找不到確切定義，明確告知使用者並詢問，禁止自行推測填補
> 4. 只有查證完成後才開始產出
> 5. 產出時引用的每個專案特定值，必須能指出來源文件位置

#### 建議 2：Uria 維持抽查習慣

每當 Claude 產出涉及專案特定代碼的文件，Uria 收到後先隨機抽查 3 個值對照規則書。本次事故正是這個習慣攔下的。

## 七、給未來 Claude 實例的備註

### 如果你接手本專案

請閱讀此段並內化為你的工作紀律。

**第一條：產出前先讀。**

你看到這份專案有完整的規則書、支柱文件、MOD 指令、狀態紀錄。**這不是裝飾**。如果你要為這個專案產出任何東西，動筆前必須先讀完相關章節。不是掃過去，是讀完。

**第二條：不要憑印象填專案特定值。**

本專案有自己的陣營系統（八陣營極）、自己的稀有度系統、自己的家族代碼、自己的位階代碼。這些不是通用 TCG 常識能推斷的。如果你不確定某個值，**查**；查不到，**問**；絕對不要**猜**。

**第三條：知道文件存在 ≠ 讀過文件。**

這是本事故的核心失敗。如果你發現自己在寫東西時「大概記得規則書裡有這個概念」，這就是警訊——立刻停手去查，不要繼續往下寫。

**第四條：技術細節不該問 Uria。**

Uria 明確表示自己是設計師不是工程師。埠號、API 路徑、檔案結構、環境變數這些問題，請自行查專案檔案判斷。若真的無法判斷，應說明「我判斷這樣合理，若不符合你的環境請指正」，而不是直接丟問題給 Uria。

**第五條：犯錯後不要自己重做。**

如果 Uria 發現你的產出有大量錯誤，**不要說「讓我重做一份」**。同一個 Claude 實例重做會再犯類似錯誤。正確做法是列出錯誤清單，交給 Claude Code 或下一個聊天室處理。

### 本事故的具體教訓

這六份文件中，**幾乎每一個錯誤都是一次搜尋或一次 `view` 就能避免的**。加總起來 15 分鐘就能做完。Claude 省了 15 分鐘，讓 Uria 多花了好幾個小時處理後果。

你如果接手這個專案，請把這 15 分鐘視為**不可壓縮的成本**，不是可以跳過的形式。
