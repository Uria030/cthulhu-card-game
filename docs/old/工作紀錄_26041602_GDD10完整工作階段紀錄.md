### 2.2 關鍵設計取捨與決議

**取捨一：`encounter_definitions` 擴充或獨立**

Claude 提供路線 A（擴充既有表）與路線 B（獨立建表）的分析。最終選擇路線 B — 獨立建 `monster_templates`、`monster_variants`、`monster_attack_cards` 等表。

**理由：** 怪物設計的複雜度（家族系統、位階變體、招式池、詞綴、狀態體系、AI 行為）遠超出通用遭遇表能承載的範圍。

**決議二：怪物個體設計而非模板**

Uria 明確要求怪物可以「一隻一隻的做好」，不是單純的模板。確立五層結構：

```
怪物家族（10 組）
  └─► 怪物種族（基礎設計稿）
        └─► 位階變體（具體可召喚個體）
              ├─► 招式池
              └─► 敘事狀態描述
```

以深潛者為例：
- 「深潛者」是種族（一個基礎設計稿）
- 「深潛者（雜兵）」「深潛者精英」「深潛者長老」各自是獨立變體
- 每個變體有自己的招式池和敘事狀態描述

**決議三：場景設計器只決定家族池**

場景設計器不指定具體怪物。場景編輯器（MOD-07）決定的是「這關開放哪幾個家族池」。城主 AI 在遊戲運行時從開放的家族池中自行決定召喚哪隻具體怪物。

**決議四：招式池兩層結構**

每隻怪物的招式來自：
- **種族級共享招式**：該種族下所有變體都可使用
- **變體專屬招式**：只有該變體可使用

### 2.3 草案 v2 的完整內容採納

Uria 提供的《怪物家族設計草案 v2》已完整定義：

- 7 個主神家族：克蘇魯、哈斯塔、莎布、奈亞、猶格、克圖格亞、伊格
- 每個家族的戰鬥特色、傷害偏重、戰鬥節奏、典型詞綴、AI 偏好
- 施放的負面狀態、使用的正面狀態、狀態免疫
- 元素弱點、抗性、**神秘無抗性**（絕對規則）
- 恐懼特色、敵對家族
- 每家族的位階範圍與具體生物

加上名詞對照表原有的 3 個類型（凡人墮落者、亡者回響、獨立存在），共 **10 組家族**。另預留 3 個擴充家族（札特瓜、修德·梅爾、諾登斯）。

**提問一：連接關係歸屬**

**提問三：隱藏調查點粒度**

**提問四：`locations` 表結構調整**

現有 Schema 中 `locations` 掛在 `scenarios` 底下，需要拆出來獨立。Uria 確認方向正確。

**提問五：翻面條件**

Claude 以實體桌遊的「翻面」邏輯提出分層方案。

**Uria 的關鍵反饋：** 實體桌遊需要翻面是因為物理卡片的限制，但這是電子遊戲不需要這樣。

這個反饋改變了地點的資訊揭露機制設計：
- 不設「正面/背面」欄位
- 改為「可見資訊」和「隱藏資訊」兩組內容
- 隱藏資訊有各自的揭露條件，達成時直接顯示給玩家

**提問六：地點不是固定形狀**

Uria 明確提出：地點不是固定形狀的，可以是任何形狀。

Claude 解讀為三個層面：視覺呈現、連接關係、尺度。Uria 確認三個層面都涵蓋。

這個原則的具體意涵：
- 視覺呈現：地點的視覺素材可以是任何形狀（不限矩形卡片）
- 連接關係：地點可連到任意數量的鄰居，關卡編輯器決定
- 尺度：地點可代表任何規模的空間，尺度標籤為可選欄位而非必填

### 4.2 SVG 繪製功能的討論

Uria 最初說「SSL 繪製功能」，Claude 猜測是 SVG 的筆誤。Uria 確認是 SVG。

**Uria 提供的三張實體桌遊參考圖：**
- 圖一：Atrio 中庭（彩色擬真俯視圖，最高成品級）
- 圖二：紅黑棋盤圓形中庭（幾何風格，中等細節）
- 圖三：Fitzgerald Mansion（黑白手繪平面圖，純示意圖）

**Uria 提供的三張不同尺度參考圖：**
- 圖一、圖二：Arkham LCG 桌遊的地點卡片呈現
- 圖三：關卡層級的地圖配置示意（不同關卡用不同數量卡片組成）

**關鍵結論：**
- 地點的尺度越大，內部細節越不需要
- 房間級：有家具、門窗、地板花紋
- 街區級：只有建築物輪廓和街道
- 城市級：只是一個建築立面或地標圖示
- 跨國級：一張地標照片或標誌性符號

**測試版 SVG 生成的範圍決議：**
- 僅限室內俯視平面圖
- 風格：純黑白線稿 + 文字標籤（類似 Fitzgerald Mansion）
- 中尺度（街區/城市）和大尺度（城鎮/國家）留待後期階段
- 先提供圖片上傳欄位，讓設計師用靜態圖片代替大尺度地點

### 4.3 技術方向確認

Uria 問：「我可以整個遊戲畫面都用 SVG 做出來嗎？」

Claude 回應：技術上可以但有取捨。
- SVG 擅長：幾何圖形、線條、UI 框架、圖示、資訊性示意圖
- SVG 不擅長：擬真插畫、細緻光影、複雜紋理
- 業界主流做法：混合使用 — SVG 處理 UI 和地圖，點陣圖處理立繪和插畫

**最終方向確認：** 測試版用純 SVG，正式版再補點陣圖。Uria 確認方向正確。

## 五、MOD-10 城主設計器完整設計過程

### 5.1 定位確認

Uria 說明城主設計器有點像卡片設計器，管理兩種卡片：

**神話卡（Mythos Cards）：**
- 城主的「手牌」
- 上面記載花費的動作點數（越強大越貴）
- 有發動時機（調查員階段響應 / 敵人階段 / 兩者皆可）

**遭遇卡（Encounter Cards）：**
- 不同地點風格的神秘事件
- 讓調查員做出選擇、做出檢定

### 5.2 五項規格確認

**問題一：神話卡效果結構**

Uria 答：神話卡需要結構化的效果語言。雖然玩家看不到，但動作種類是固定的。

實作決議：仿照 MOD-01 卡片設計器的 `card_effects` 模式，為神話卡設計一套動作代碼。

Claude 列出 18 個動作代碼：
- 召喚類：`summon_monster`、`spawn_at_location`
- 議程類：`advance_agenda`、`reveal_act`
- 環境類：`environment_change`、`disconnect_location`
- 狀態類：`inflict_status`、`remove_buff`
- 全場類：`damage_all`、`force_check_all`
- 混沌袋類：`modify_chaos_bag`
- 遭遇牌堆類：`draw_encounter`、`shuffle_encounter_deck`
- 響應取消類：`cancel_player_action`、`force_reroll`
- 敘事類：`narrative_only`、`set_flag`

**問題二：遭遇卡選項數量**

Uria 答：2–3 個選項，然後再講一段故事。

實作決議：
- 每張遭遇卡 2–3 個選項
- 每個選項有情境檢定 + 成功/失敗的尾聲敘事
- 「故事後半段」讓玩家感受選擇重量

**問題三：城主行動點公式**

Uria 答：每多一個玩家多兩點，其他先這樣。

最終公式（儲存在 `game_balance_settings`）：
- 難度 1–5 對應基礎點數 2/3/4/5/6 點
- 人數加成：每多一人 +2 點（從第 2 人起算）
- 可跨回合累積

**問題四：神話卡與怪物召喚的整合**

Uria 答：正確，會可以多付點數讓怪物升階。

Claude 回應：怪物升階點數不是由每張神話卡決定，而是在全域設定中統一定義。

Uria 確認：對，升階的點數成本不是在神話卡層級決定。

最終決議：
- 神話卡的 `summon_monster` 動作只指定家族池 + 召喚數量 + 基礎位階
- 升階成本放在 `game_balance_settings`（雜兵→威脅 +2、威脅→精英 +3 等）
- 城主在遊戲運行時，根據當下點數自行決定是否升階

**問題五：拆檔計畫**

Uria 答：照 Claude 建議即可（拆 3 份）。

### 5.4 MOD-10 的關鍵設計原則

1. **神話卡是城主的武器，玩家看不到內容**：設計時重點在機制準確而非玩家閱讀友善

## 八、關鍵工作原則歸納

### 8.1 Uria 的工作偏好

1. **規格先確認方向再動筆** — 禁止重工浪費 token
2. **所有產出檔案太大時拆解成 3+ 份交付**
3. **Seed Data 必須預填規則書文字** — 讓管理員打開設計器即可看到內容
4. **技術決策不介入** — 但會問「擴充跟不擴充的差別是什麼」來做判斷
5. **明確反饋錯誤方向** — 不會讓 Claude 繼續錯誤路徑
6. **使用 DEBUG 迴圈** — 收到修正提醒立刻停止，先分析後修正

### 8.2 Claude 在本聊天室的收穫

1. **第一次被提醒時就信任使用者的指向** — 搜索工具有盲區
2. **電子遊戲不是實體桌遊的數位化** — 翻面、卡片形狀等限制不需要繼承
3. **規格提案要明確標記「這是現有規則」vs「這是新產生的設計」**
4. **拆檔不是按篇幅拆，是按邏輯拆**（資料庫/API vs 介面 vs AI/總覽）
5. **繼承鏈（家族→種族→變體）和共享結構（種族級招式 vs 變體專屬）是可以共用的設計模式**

### 8.3 本聊天室建立的重要設計模式

**五層結構模式（MOD-03）：**
```
家族 → 種族 → 變體 → 招式池/敘事
```

**三欄佈局模式（通用）：**
```
左側：導航選擇 + 列表
中央：分頁標籤頁
右側：編輯區塊
```

**結構化動作語言模式（MOD-10）：**
```
action_code（枚舉）+ action_params（JSONB）+ description（人類可讀文字）
```

**全域平衡設定模式（MOD-10）：**
```
game_balance_settings 表集中管理跨系統的平衡參數
```
