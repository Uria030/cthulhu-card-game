## 三、關鍵設計決議總整理

### 3.1 支柱五 §1.2 陣營主屬性修正案（重大）

**錯誤狀態（支柱五 v0.1 原文）：**

| 陣營 | 主屬性 | 問題 |
|------|--------|------|
| N 天啟 | 意志 | — |
| F 聖燼 | 意志 | 與 N 重複 |
| 其他陣營 | — | 力量無人對應 |

**修正後（確認定案）：**

| 陣營 | 主屬性 | 備註 |
|------|--------|------|
| E 號令 | 魅力 | 領導者靠社交影響力 |
| I 深淵 | 智力 | 與 T 共享的一極（刻意設計） |
| S 鐵證 | 感知 | 實證派靠觀察 |
| N 天啟 | 意志 | 法術施放、精神防禦 |
| T 解析 | 智力 | 與 I 共享 |
| **F 聖燼** | **力量** | **（修正）燃燒肉身的魄力，消防員原型** |
| J 鐵壁 | 體質 | 肉體堡壘 |
| P 流影 | 敏捷 | 在縫隙中穿梭 |

**設計原理：** 七屬性各對應一個陣營極。八個極對應七個屬性必有一個共享，共享點落在智力上（I 與 T）。

### 3.2 規則書第六章 §2.2 創角總點數修正（重大）

| 項目 | 原值（錯） | 修正後 |
|------|-----------|--------|
| 創角總點數 | 21 | **18** |

**配點結構：**
- 基礎 7 點（每屬性 +1）
- 主陣營主屬性 +3
- 三個副陣營主屬性各 +1（共 +3）
- 玩家自由分配 5 點
- **總計 7 + 3 + 3 + 5 = 18**

**差額來源：** 原 21 點的差額 3 點，透過天賦樹的 5 次屬性提升（2/4/7/9/11 級各 +1）在冒險中補回並超越。

### 3.3 64 預設調查員模板設計框架（新創）

**結構：**
- 16 MBTI 四字碼 × 4 職業 = **64 個預設模板**
- 職業序號 1–4 對應四字碼的字母位置（例：ENTJ-2 偏重第 2 字母 N）
- 每個模板有 `dominant_letter` 欄位標記偏重字母

**設計哲學：**
- 預設模板骨架預先存在於資料庫，但所有敘事欄位**留空**
- 由設計者以設計器逐一填寫，AI 僅輔助敘事
- AI 生成結果不自動寫入 DB，設計者最終決定

**陣營等級機制（Lv1/Lv2）預留：**
- 陣營等級系統尚未定案，為**待定設計空間**
- MOD-11 的 Schema 已預留欄位，但不實作邏輯

### 3.4 調查員強度 V 值量化系統（新創）

**定義：** 1V = 玩家花一個行動可以做到的事情的收益量。

**三個維度：**

| 維度 | 代碼 | 計算方式 |
|------|------|---------|
| A 素質 | `baseline_value` | 屬性 V + HP V + SAN V |
| B 能力 | `ability_value` | 戰鬥熟練 V + 能力文字 V |
| C 簽名卡 | `signature_total_value` | 所有簽名卡的打出 V 加總 |
| 弱點（扣減） | `weakness_value` | 負值，= effect × prob × rounds |

**核心係數（Uria 確認）：**

| 項目 | V 值係數 |
|------|---------|
| 每點屬性 | **2.0V**（每屬性 4 次檢定 × 0.5V） |
| HP 上限每點 | **1.5V**（恢復 HP 基準） |
| SAN 上限每點 | **1.5V**（恢復 SAN 基準） |
| 戰鬥熟練每個 | **7.5V**（+1 × 15 次戰鬥檢定 × 0.5V） |

**分級機制（Uria 選擇「不設固定目標」）：**
- 用 z-score 對實際平均與標準差偵測離群值
- z > ±2.0：紅色警告（`overpowered`/`underpowered`）
- z > ±1.5：黃色警告（`above_average`/`below_average`）
- 未完成模板標為 `incomplete`

**範圍限定：**
- **比較範圍：** 素質、能力、簽名卡、弱點
- **不比較：** 起始牌組（一般卡，玩家可修改）

## 四、架構修正通告：Gemini API 整合方式

### 4.1 架構規範（重申）

**既有架構（從 MOD-01 卡片設計器起一貫使用）：**

- Gemini API 為**前端直接呼叫**
- 使用者在瀏覽器端自帶 Gemini API Key
- API Key 儲存於 `localStorage`，key 名稱：`gemini_api_key`
- 呼叫端點：`https://generativelanguage.googleapis.com/...`
- 使用模型：**Gemini 2.5 Flash**
- 各設計器**共用同一個 API Key**，不需個別設定

### 4.2 影響的指令文件（錯誤沿用架構，需更正）

本聊天室寫的以下章節**錯誤沿用後端轉發架構**，實作時需改為前端直接呼叫：

| 檔案 | 受影響章節 | 需修正內容 |
|------|----------|-----------|
| MOD-11 Part 3 | Gemini 敘事生成（4 模式 A/B/C/D） | 後端端點 `/api/admin/gemini/generate-investigator-narrative` 應廢除，改為前端直接呼叫 |
| MOD-11 Part 4 | Gemini V 值估算（3 模式 E-1/E-2/E-3） | 後端端點 `/api/admin/gemini/estimate-*` 應廢除 |
| MOD-09 Part 2 | Gemini 敘事輔助（3 模式） | 同上原則，前端直接呼叫 |

### 4.3 正確的實作方式（供 Claude Code 參考）

**前端呼叫範例（遵循 MOD-01 起的既定架構）：**

```javascript
async function callGemini(systemPrompt, userHint) {
  const apiKey = localStorage.getItem('gemini_api_key');
  if (!apiKey) {
    alert('請先在設定中輸入 Gemini API Key');
    return null;
  }

  const response = await fetch(
    `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent?key=${apiKey}`,
    {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        contents: [{
          role: 'user',
          parts: [{ text: systemPrompt + '\n\n' + userHint }]
        }],
        generationConfig: {
          temperature: 0.85,
          maxOutputTokens: 800,
          responseMimeType: 'application/json'
        }
      })
    }
  );

  const data = await response.json();
  return JSON.parse(data.candidates[0].content.parts[0].text);
}
```

**Prompt 本體不變** — MOD-11 Part 3/Part 4 與 MOD-09 Part 2 中撰寫的系統 Prompt 內容完全保留，只是「呼叫機制」從後端轉發改為前端直呼。

### 4.4 本修正不影響的部分

- 所有**資料庫 CRUD 的後端 API**（Fastify + PostgreSQL）維持不變
- 所有**業務邏輯、驗證、觸發器**等後端處理維持不變
- V 值計算的 PL/pgSQL 函數、觸發器維持在後端
- 只有「對 Gemini 發送請求」這一層的實作位置改變

### 4.5 根本原因說明

此錯誤源於 Claude 撰寫 MOD-09 與 MOD-11 指令文件時，**未先查閱既有模組（MOD-01、MOD-04）的 Gemini 整合方式**，預設為後端轉發架構。實際上專案從 MOD-01 起即採用前端呼叫，相關架構已在 MOD-04 第六部分明確記載（`localStorage key: gemini_api_key`、使用 Gemini 2.5 Flash）。

未來撰寫設計器指令時，**應先查既有模組的共用架構**，避免重複沿用錯誤模式。

### 10.2 關鍵工作原則

1. **Uria 是設計師，不是工程師** — 技術決策由 Claude 自行判斷，絕不丟回給 Uria 決定
2. **嚴格執行 DEBUG 迴圈** — Uria 指正錯誤時立即停止、分析、確認、再修正
3. **產出檔案太大時拆成 3+ 份交付** — 這是 Uria 的使用者偏好
4. **撰寫新模組前務必檢視既有模組** — 特別是 MOD-01、MOD-04 的共用架構（如 Gemini 前端 API、`admin-shared.js` 等）
5. **敘事留白哲學** — 預設資料的敘事欄位留空，AI 只輔助不自動寫入
6. **台灣繁體中文 + TRPG 慣用譯名** — 避免中國大陸用語、避免自創翻譯

### 10.3 已知易犯的錯誤模式（本聊天室歸納）

1. 未查既有模組就動筆，導致架構不一致
2. 把 Uria 簡短指示當成「要我做選擇」而過度提問技術決策
3. 過度消除設計空間（把「待定」解讀為「不存在」）
4. 沿用錯字為專有名詞
5. 不主動要求確認，容易跳步推論

### 10.4 本聊天室建立的重要設計模式（可供其他模組參考）

**模式一：骨架預建 + 敘事留白**
- MOD-11 的 64 模板骨架（僅結構資料，敘事全空）
- 適用場景：有明確枚舉上限的設計對象（四字碼 × 職業、狀態 × 家族等）

- 適用場景：需要跨設計對象做強度平衡的系統

**模式三：三階級模式的詞條設計**
- MOD-09 的 scaling / fixed / choice
- 適用場景：效果有「漸進」「固定」「多選」三種型態的模組
