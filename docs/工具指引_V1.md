## §A 工具清單與能力摘要

### A.1 MCP Server（全域註冊，所有專案可用）

#### Context7（Upstash）

- **能力：** 從官方文件源即時拉取最新的、版本對應的 library 文件和程式碼範例，注入到 prompt 中
- **調用方式：** 在 prompt 中加入 `use context7` 即可自動觸發
- **適用場景：**
  - 使用 Bootstrap 5、SheetJS、pptxgenjs、xlsx-js-style 等第三方 library 時
  - 對 API 用法沒有把握、或懷疑訓練資料可能過時時
  - 寫新模組需要查詢 library 的最新 API 或版本差異時
- **禁止場景：** 查詢 GAS 原生 API（Google 文件通常不在 Context7 收錄範圍，直接用訓練知識即可）

#### GitHub MCP（GitHub 官方）

- **能力：** 直接操作 GitHub repository — 瀏覽檔案、管理 issue、建立/審查 PR、查看 diff
- **調用方式：** Claude Code 可直接使用 GitHub 相關工具函式
- **適用場景：**
  - 查看 commit 歷史、比對版本差異
  - 建立或更新 issue 追蹤 bug/feature
  - 小白與小黑之間的程式碼同步確認
- **禁止場景：** 不要用 GitHub MCP 來讀本地檔案（直接用檔案系統）

### A.2 CLI 工具（已安裝，透過終端機呼叫）

#### MarkItDown（Microsoft 官方）

- **能力：** 將 PDF、Word、PowerPoint、Excel、圖片（含 OCR）、HTML、CSV 等檔案轉換為結構化 Markdown
- **調用方式：** `python -m markitdown <檔案路徑> -o <輸出路徑.md>`
- **適用場景：**
  - 需要讀取合約書 PDF、預算書 Excel、投標文件 Word 時
  - 任何非純文字檔案需要讓 AI 理解內容時
  - pptxgenjs 產出簡報後的文字內容驗證（`python -m markitdown output.pptx`）
- **重要：** Windows 環境統一使用 `python -m markitdown`，不要直接呼叫 `markitdown`

### A.3 Skills（安裝在專案 `.claude/skills/` 下）

#### UI-UX-Pro-Max（設計智能資料庫）

- **能力：** 提供 50+ UI 風格、161 色彩配置、57 字體配對、99 UX 準則的可搜尋資料庫
- **調用方式：** `python .claude/skills/ui-ux-pro-max/scripts/search.py "<關鍵字>" --domain <domain>`
- **可搜尋 domain：** styles、colors、typography、charts、ux、products
- **適用場景：**
  - 設計新模組的視覺風格時，搜尋靈感和配色方案
  - 選擇字體配對
  - 查詢特定設計模式的最佳實踐
- **與 Impeccable 的分工：** UI-UX-Pro-Max 負責「設計前的靈感探索」，Impeccable 負責「設計後的品質把關」

#### Impeccable（前端設計品質守門員，18 個 slash commands）

- **能力：** 偵測並修正 AI 產出的前端設計反模式（AI slop），18 個專業化 slash commands
- **核心 commands：**
  - `/audit` — 無障礙、效能、響應式品質檢查
  - `/critique` — UX 審查：層次、清晰度、情感共鳴
  - `/polish` — 上線前最終審查（對齊、間距、細節）
  - `/typeset` — 排版修正
  - `/colorize` — 色彩系統優化
  - `/layout` — 佈局和間距修正
  - `/overdrive` — 大膽的視覺特效（beta）
  - `/adapt` — 跨裝置響應式適配
  - `/animate` — 有目的性的動態效果
  - `/distill` — 精簡：移除不必要的複雜度
  - `/optimize` — 效能優化
  - `/bolder` — 強化視覺衝擊力
  - `/quieter` — 降低視覺噪音
  - `/clarify` — 提升可讀性和理解度
  - `/delight` — 增加微互動和愉悅感
  - `/shape` — 視覺形狀和幾何優化
- **適用場景：** 任何前端 HTML/CSS 產出的品質驗證
- **設計脈絡：** 首次使用建議先執行 `/impeccable teach` 建立專案設計脈絡（`.impeccable.md`）

#### Web Design Guidelines（Vercel Labs）

- **能力：** 依據 100+ 條 Web Interface Guidelines 規則審查 UI 程式碼的正確性
- **調用方式：** `/web-design-guidelines <檔案路徑>`
- **審查範圍：** 無障礙（ARIA）、鍵盤支援、表單行為、動畫、效能、UX 最佳實踐
- **適用場景：** 前端程式碼的合規性檢查，特別是無障礙和效能面向
- **與 Impeccable 的分工：** Web Design Guidelines 偏「規則合規」，Impeccable 偏「美學品質」

#### Design-MD（Google Labs / Stitch）

- **能力：** 分析專案並產生 DESIGN.md 設計系統文件（色彩 token、字型比例、間距規則、元件模式）
- **調用方式：** `/design-md`
- **適用場景：**
  - 從既有專案萃取視覺設計系統
  - 建立 AI 可讀的設計規範文件，確保所有模組視覺一致
- **未來計畫：** 與設計準則 V3.6 的視覺層萃取（DESIGN.md 提取）搭配使用

## §B Claude Code 工具使用守則

> **本段落應追加至 `CLAUDE.md`。**

### B.1 主動調用原則

Claude Code 不需要等待使用者指示，應在以下情境**主動**使用對應工具：

| 情境 | 應主動使用的工具 | 具體做法 |
|------|----------------|---------|
| 使用任何第三方 library 的 API | Context7 | prompt 加入 `use context7`，查詢該 library 最新文件 |
| 需要讀取 PDF/Word/Excel 檔案 | MarkItDown | `python -m markitdown <檔案> -o <輸出.md>` 後讀取 md |
| 產出前端 HTML/CSS 後 | Impeccable `/audit` | 交付前自動執行品質檢查 |
| 產出前端程式碼涉及無障礙 | Web Design Guidelines | `/web-design-guidelines` 合規檢查 |
| 對 Git 操作有疑問或需查歷史 | GitHub MCP | 直接使用工具函式查看 diff/history |
| 新模組的視覺設計探索 | UI-UX-Pro-Max search | 搜尋風格、配色、字體推薦 |

### B.2 Context7 強制使用場景

以下 library 因版本迭代頻繁，**必須**使用 Context7 查詢最新 API 後再寫程式碼：

| Library | 原因 |
|---------|------|
| pptxgenjs | Shape Type 常數、Shadow API 格式曾因版本變更導致嚴重錯誤 |
| SheetJS / xlsx-js-style | 匯出格式、樣式 API 差異大 |
| Bootstrap 5 | Modal API、Utility classes 在不同小版本間有變化 |

以下**不需要**使用 Context7：
- Google Apps Script 原生 API（SpreadsheetApp、UrlFetchApp 等）
- LINE LIFF SDK（版本穩定，訓練資料足夠）
- RAGIC REST API（非公開收錄，依設計準則 §14 為準）

### B.3 MarkItDown 使用時機

```bash
# 讀取使用者提供的外部文件
python -m markitdown "D:\gas\docs\合約書.pdf" -o "D:\gas\docs\合約書.md"

# pptxgenjs 產出驗證（參照 PptxGenJS 簡報製作準則 §4.3）
python -m markitdown output.pptx

# Excel 預算書結構確認
python -m markitdown "預算書樣本.xlsx" -o "預算書結構.md"
```

### B.4 前端品質檢查流程（交付前必做）

產出或修改前端 HTML/CSS 檔案後，依序執行：

```
1. /audit          ← 無障礙 + 效能 + 響應式基本檢查
2. /polish         ← 對齊、間距、細節微調
3. /web-design-guidelines <檔案>  ← 合規性最終確認
```

若為新模組的視覺設計，在寫程式碼之前先執行：

```
1. python .claude/skills/ui-ux-pro-max/scripts/search.py "<風格關鍵字>" --domain styles
2. python .claude/skills/ui-ux-pro-max/scripts/search.py "<色彩關鍵字>" --domain colors
3. 根據搜尋結果確認設計方向後再開始實作
```

### B.5 規格文件中的工具標註格式

claude.ai 撰寫規格文件交付給 Claude Code 時，應在相關步驟標註工具使用指示。標準格式：

```markdown
> 🔧 **工具指示：** 使用 Context7 查詢 pptxgenjs 最新的 addShape API 格式

> 🔧 **工具指示：** 使用 MarkItDown 將附件預算書.xlsx 轉為 Markdown 後讀取結構

> 🔧 **交付前檢查：** 執行 /audit + /polish + /web-design-guidelines
```

### B.6 不適用場景提醒

以下情境**不應**使用這些工具，避免浪費時間或產生干擾：

| 不應使用 | 原因 |
|---------|------|
| Context7 查 GAS API | Google 官方文件不在收錄範圍 |
| Context7 查 RAGIC API | RAGIC 為非主流平台，依設計準則為準 |
| Impeccable 審查後端 Code.gs | Impeccable 專為前端 HTML/CSS 設計 |
| UI-UX-Pro-Max 在 bugfix 時 | 修 bug 時不需要設計探索 |
| MarkItDown 讀純文字 .md/.txt | 直接讀檔即可，不需要轉換 |
